## 16:00 店铺后台接口数据访问方案设计

**需求背景：**
- 为店铺管理员后台配置接口数据访问
- 只有店铺助手能访问，芋圆（主Agent）不能访问
- 考虑使用 MCP 模式或其他方案

**方案对比：**

### 方案1: MCP Server 模式

**原理：**
- 为店铺后台写 MCP Server（标准化接口服务）
- 店铺助手通过 MCP 协议连接
- 芋圆没有配置这个 MCP Server，所以无法访问

**配置示例：**
```yaml
# shopbot.yaml
mcpServers:
  shop-admin:
    command: npx
    args: ["-y", "@your-org/shop-mcp-server"]
    env:
      SHOP_API_KEY: ${SHOP_API_KEY}
      SHOP_BASE_URL: https://admin.yourshop.com/api
```

**优点：**
- ✅ 标准化，可复用
- ✅ 隔离性好，芋圆完全看不到
- ✅ 支持复杂的数据操作

**缺点：**
- ⚠️ 需要开发 MCP Server（工作量）
- ⚠️ 需要维护额外的服务

---

### 方案2: 自定义 Tools（推荐）

**原理：**
- 在店铺助手的 workspace 中配置专用工具
- 工具直接调用店铺后台 API
- 通过环境变量控制访问权限

**配置示例：**
```yaml
# shopbot.yaml
tools:
  shop:
    api:
      baseUrl: https://admin.yourshop.com/api
      headers:
        Authorization: Bearer ${SHOP_API_TOKEN}
    
    operations:
      - name: getInventory
        endpoint: GET /inventory
        description: "查询库存"
      
      - name: updatePrice
        endpoint: POST /products/{id}/price
        description: "修改商品价格"
      
      - name: getOrders
        endpoint: GET /orders
        description: "查询订单"

# 环境变量（只有 shopbot 能读到）
# ~/.openclaw/workspace-ecommerce/.env
SHOP_API_TOKEN=sk_live_xxxxxxxxxxxx
SHOP_BASE_URL=https://admin.yourshop.com/api
```

**优点：**
- ✅ **无需开发额外服务**，配置即可用
- ✅ 隔离性好，芋圆无法访问 shopbot 的 workspace
- ✅ 灵活，可根据需求随时添加新操作
- ✅ 与 OpenClaw 生态整合最好

**缺点：**
- ⚠️ 需要手动配置 API 映射
- ⚠️ 复杂逻辑可能需要写脚本（但可以用 subagent 解决）

| **维护成本** | 高（需维护额外服务） | 低 |

| **适用场景** | 复杂数据操作，长期维护 | 快速上线，简单够用 |
