### 机制澄清：Sub-Agent vs 平行 Agent (10:30)

**哥哥指出**: 混淆了两种机制，需要分清楚

**本质区别**:
| 维度 | Sub-Agent (sessions_spawn) | 平行 Agent (Multi-Agent Routing) |
|------|---------------------------|----------------------------------|
| 生命周期 | 临时任务，完成即销毁 | 永久配置，长期存在 |
| 创建方式 | 运行时动态 spawn | 配置文件预定义 |
| Session | 独立临时会话 | 独立持久会话 |
| 适用场景 | 一次性后台任务 | 固定角色分工 |
| 开销 | 低（用完即弃） | 高（持续占用资源）|

**任务类型 → 执行策略映射**:
1. **日常对话类** → Main 自己处理
2. **专业技能类**（编程/视频）→ 平行 Agent（需持久 Session）
3. **信息收集类**（搜索/调研）→ Sub-Agent
4. **复杂协作类** → 平行 Agent 协作或云端协作
5. **紧急/临时类** → Sub-Agent

**架构决策 5 问**:
1. 任务是否频繁发生？→ 是：平行 Agent
2. 是否需要持久 Session？→ 是：平行 Agent
3. 是否需要固定角色/技能？→ 是：平行 Agent
4. 任务是否可并行？→ 是：多 Sub-Agent 并行
5. 是否需要跨设备协作？→ 是：云端 + 本地平行 Agent

**渐进策略**:
- Phase 1（观察期 1-2 周）：保持单 Agent，用 Sub-Agent，记录任务分布
- Phase 2（分析期）：统计高频任务，评估平行 Agent ROI
- Phase 3（决策期）：数据驱动决策（编程>30%→Coder，视频>20%→Video）

**立即执行**:
1. ✅ 保持当前单 Agent 配置
2. ✅ 开始记录任务类型分布（memory/task-distribution.json）
3. ✅ 每周汇报统计数据

**项目**: https://github.com/snarktank/antfarm

**三条核心工作流**:
| 工作流 | Agent 数 | 用途 | 输出 |

**安装方式**:

**依赖要求**:
- Node.js >= 22
- OpenClaw v2026.2.9+
- gh CLI（创建 PR）

**集成建议**（混合模式）:

**根因**: 飞书双 Bot 配置 + bindings 未指定 account，导致同一条消息被两个 Bot 同时处理

**配置问题**:
```json
// 2 个 Bot 账号
"accounts": {
  "main": { "appId": "cli_a909aafbc3f99cd6", "botName": "芋圆" },
  "video": { "appId": "cli_a928956164b8dbdb", "botName": "视频助手" }
}

// bindings 匹配所有 account（未指定）
"bindings": [
  { "agentId": "video-commerce", "match": { "channel": "feishu" } }
]
```

**执行流程**:
哥哥发消息 → main Bot 收到 + video Bot 收到 → 两个 Bot 都回复 → 哥哥收到 2 条相同消息

**修复方案**（方案 A - 推荐）:
1. 删除 `channels.feishu.accounts.video` 配置
2. 保留 `main` 账号
3. 重启 Gateway
